1Descripción del Sistema

1.1 ¿Qué problema resuelve?

En consultorios que gestionan turnos por teléfono se repiten cuatro problemas:

  • La línea telefónica colapsa en las primeras horas de atención.
  • Los registros en papel generan dobles reservas.
  • El profesional no conoce su agenda hasta llegar al consultorio.
  • No existe historial digital de cancelaciones ni trazabilidad.

VanguardiaShift digitaliza el proceso completo. El paciente reserva desde el navegador en cualquier momento y el sistema valida que el horario esté libre antes de confirmarlo.

1.2 Roles del sistema

RolAcciones permitidasRestricción
ClienteReservar turnos, ver disponibilidad, cancelar sus propios turnosNo puede ver ni modificar turnos de otros usuarios
ProfesionalVer su agenda, completar o cancelar sus turnosSolo gestiona sus propios turnos
AdminAcceso completo al sistema, gestión de usuarios—

1.3 Objetivos del sistema

  1. Eliminar las reservas telefónicas para consultorios de hasta 10 profesionales.
  2. Impedir que un mismo horario se reserve simultáneamente por dos personas (control de concurrencia).
  3. Ofrecer agenda en tiempo real para el profesional desde cualquier dispositivo.
  4. Mantener historial completo e inalterable, incluidos los turnos cancelados.
  5. Implementar autenticación con tokens revocables y autorización estricta por rol.
📘 Concepto: Control de concurrencia

Es la capacidad de un sistema para manejar múltiples operaciones simultáneas sin que se generen inconsistencias. En este caso, evita que dos usuarios reserven el mismo horario al mismo tiempo.

2Arquitectura Propuesta

2.1 Diseño desacoplado

El frontend (interfaz de usuario) y el backend (servidor) son proyectos independientes. No comparten código ni servidor. Se comunican exclusivamente mediante el protocolo HTTP, enviando y recibiendo datos en formato JSON.

2.3 ¿Qué significa "desacoplado"?

En una arquitectura tradicional (acoplada), el frontend y el backend viven en el mismo servidor y comparten código. En una arquitectura desacoplada:

  • El frontend es una aplicación independiente que corre en el navegador del usuario.
  • El backend es una API independiente que corre en un servidor remoto.
  • Ambos se comunican mediante peticiones HTTP con formato JSON.
  • Cada uno puede cambiar o escalar sin afectar al otro.
📘 Concepto: API REST

API (Application Programming Interface) es un conjunto de reglas que permite que dos sistemas se comuniquen. REST (Representational State Transfer) es un estilo de diseño que usa URLs para identificar recursos y métodos HTTP para operar sobre ellos: GET para leer, POST para crear, PUT para actualizar, DELETE para eliminar.

📘 Concepto: JSON

JSON (JavaScript Object Notation) es un formato de texto para representar datos estructurados. Es legible por humanos y parseable por máquinas. Ejemplo: {"nombre": "Carlos", "edad": 30}

2.4 Flujo de comunicación

Cuando un usuario reserva un turno, ocurre este flujo:

  1. El usuario selecciona un médico en el frontend (Vue.js).
  2. Vue.js envía una petición GET /api/profesionales al backend (Laravel).
  3. Laravel consulta la base de datos (MySQL) y devuelve la lista en formato JSON.
  4. Vue.js recibe el JSON y muestra los médicos en pantalla.
  5. El usuario selecciona fecha y hora.
  6. Vue.js envía GET /api/profesionales/{id}/disponibilidad?fecha=YYYY-MM-DD.
  7. Laravel consulta qué horarios están ocupados y devuelve los disponibles.
  8. El usuario confirma la reserva.
  9. Vue.js envía POST /api/turnos con los datos del turno.
  10. Laravel ejecuta lockForUpdate() dentro de una transacción para verificar disponibilidad.
  11. Si el horario está libre, Laravel inserta el turno en MySQL y responde con HTTP 201.
  12. Vue.js muestra la confirmación al usuario.

📘 Concepto: Clave foránea (Foreign Key)

Es una columna en una tabla que referencia la clave primaria de otra tabla. Establece una relación entre registros. En este caso, cliente_id y profesional_id en la tabla turnos referencian id en la tabla users. Esto garantiza que un turno siempre esté asociado a usuarios existentes.

📘 Concepto: Índice compuesto

Es un índice de base de datos que abarca múltiples columnas. En la tabla turnos, el índice (profesional_id, fecha_hora) optimiza la consulta más frecuente del sistema: "¿hay algún turno para este profesional en este horario?". Sin este índice, la base de datos recorrería toda la tabla (full scan). Con el índice, va directamente al registro.

3Frontend — Vue.js 3

3.1 ¿Qué es Vue.js 3?

Vue.js es un framework de JavaScript para construir interfaces de usuario. La versión 3 introduce la Composition API, que permite organizar el código por funcionalidad en lugar de por tipo de elemento (datos, métodos, propiedades computadas).

📘 Concepto: Composition API vs Options API

Options API (Vue 2): Organiza el código en bloques separados: data, methods, computed, mounted. Cada funcionalidad del sistema queda dispersa en múltiples bloques.

Composition API (Vue 3): Organiza el código por funcionalidad. Todo lo relacionado con "reserva de turnos" (estado, métodos, propiedades) vive junto en el mismo bloque de código. Esto facilita la lectura y el mantenimiento.

3.3 Flujo de reserva — 3 pasos

Paso 1: Seleccionar profesional

Al montarse el componente, se ejecuta onMounted() que llama a cargarProfesionales(). Esta función hace una petición GET /api/profesionales al backend y recibe un array de objetos JSON con los datos de cada médico.

// BookingCalendar.vue — Paso 1
const cargarProfesionales = async () => {
  const response = await api.get('/api/profesionales')
  profesionales.value = response.data.data
}

Paso 2: Seleccionar fecha y horario

Al elegir una fecha, se ejecuta cargarDisponibilidad() que consulta GET /api/profesionales/{id}/disponibilidad?fecha=YYYY-MM-DD. El backend responde con los horarios libres y ocupados. Los ocupados se muestran deshabilitados visualmente.

    `/api/profesionales/${profesionalId}/disponibilidad`,
    { params: { fecha: fechaSeleccionada.value } }

  horariosDisponibles.value = response.data.horarios_disponibles

Paso 3: Confirmar reserva

Al confirmar, se envía POST /api/turnos con el payload que incluye profesional_id, fecha_hora y motivo. El backend responde con códigos HTTP específicos:

  • 201 Created — Turno reservado exitosamente.
  • 409 Conflict — El horario fue tomado por otro usuario mientras tanto.
  • 422 Unprocessable Entity — Los datos enviados no pasaron la validación.
  • 401 Unauthorized — El token de sesión expiró o es inválido.

3.4 Servicio api.js — Axios con interceptores

Axios es un cliente HTTP que permite hacer peticiones al backend. El archivo api.js centraliza la configuración:

  • baseURL: Define la URL del backend para no repetirla en cada llamada.
  • Interceptor de request: Agrega automáticamente el token Bearer en el header Authorization de cada petición.
  • Interceptor de response: Maneja errores globales: si recibe 401, redirige al login; si recibe 403, muestra advertencia; si recibe 500+, loguea el error.
📘 Concepto: Interceptor

Es una función que se ejecuta automáticamente antes o después de cada petición HTTP. Permite modificar la petición (ej: agregar headers) o la respuesta (ej: manejar errores) de forma centralizada, sin repetir código en cada componente.

4.3 Decisiones clave de diseño

Responsabilidad Única (S de SOLID)

Cada clase tiene una sola tarea:

  • Las rutas enrutan: definen qué URL llama a qué controller.
  • El StoreTurnoRequest valida: verifica que los datos cumplan reglas antes de que lleguen al controller.
  • El TurnoController decide: ejecuta la lógica de negocio.
  • Los modelos mapean la DB: representan tablas como objetos PHP.

Un cambio en las reglas de validación no toca el controller. Un cambio en la lógica de negocio no toca la validación.

📘 Concepto: SOLID — Responsabilidad Única

Es el primer principio de SOLID. Establece que una clase debe tener una única razón para cambiar. Si una clase hace demasiadas cosas, cualquier modificación puede romper funcionalidades no relacionadas. Separar responsabilidades reduce el acoplamiento y facilita el mantenimiento.

Concurrencia — lockForUpdate()

La operación más crítica del sistema es la reserva de un turno. Sin protección, dos usuarios podrían reservar el mismo horario simultáneamente:

// TurnoController.php — store()
public function store(StoreTurnoRequest $request): JsonResponse
{
    $turno = DB::transaction(function () use ($request) {
        
        // Bloquear la fila mientras se verifica disponibilidad
        $ocupado = Turno::where('profesional_id', $request->profesional_id)
            ->where('fecha_hora', $request->fecha_hora)
            ->whereIn('estado', ['reservado', 'confirmado'])
            ->lockForUpdate()
            ->exists();

        if ($ocupado) {
            abort(409, 'El horario ya no está disponible.');
        }

        return Turno::create([...]);
    });

    return response()->json([...], 201);
}
📘 Concepto: Transacción de base de datos

Es un conjunto de operaciones que se ejecutan como una unidad indivisible. O todas las operaciones se completan exitosamente, o ninguna se aplica. Si ocurre un error en el medio, la base de datos vuelve al estado anterior (rollback). Esto garantiza la integridad de los datos.

📘 Concepto: lockForUpdate()

Es una cláusula de SQL que bloquea las filas seleccionadas para que otras transacciones no puedan modificarlas hasta que la transacción actual termine (commit o rollback). En el contexto de la reserva: cuando un usuario consulta si un horario está libre, la fila se bloquea. Si otro usuario intenta consultar el mismo horario, debe esperar. Cuando el primero termina, el segundo lee que el horario ya está ocupado.

📘 Concepto: Código HTTP 409 Conflict

Es un código de estado HTTP que indica que la petición no pudo completarse debido a un conflicto con el estado actual del recurso. En este caso, significa que el horario que el usuario intenta reservar ya fue tomado por otro usuario.

// TurnoTest.php
public function un_cliente_puede_reservar_un_turno_libre(): void
{
    // 1. Crear un cliente autenticado
    $cliente = $this->clienteAutenticado();
    
    // 2. Crear un profesional
    $profesional = $this->profesional();
    
    // 3. Enviar petición POST para reservar
    $response = $this->postJson('/api/turnos', [
        'profesional_id' => $profesional->id,
        'fecha_hora'     => now()->addDays(3)->format('Y-m-d H:i:s'),
        'motivo'         => 'Consulta de rutina',
    ]);

    // 4. Verificar respuesta HTTP 201
    $response->assertStatus(201);
    
    // 5. Verificar que el turno existe en la base de datos
    $this->assertDatabaseHas('turnos', [
        'cliente_id'     => $cliente->id,
        'profesional_id' => $profesional->id,
        'estado'         => 'reservado',
    ]);
    
    // 6. Verificar que la respuesta incluye el estado
    $response->assertJsonPath('data.estado', 'reservado');
}
📘 Concepto: assertDatabaseHas()

Es un método de PHPUnit que verifica que exista al menos un registro en una tabla de la base de datos que cumpla con las condiciones especificadas. Si no encuentra el registro, el test falla. Esto permite verificar que la operación no solo devolvió el código HTTP correcto, sino que también persistió los datos correctamente.

5.4 ¿Por qué SQLite en memoria?

Los tests verifican el comportamiento de la aplicación, no el motor de base de datos. SQLite :memory: permite que los 6 tests corran en aproximadamente 2 segundos sin servidor externo. La base se crea y destruye automáticamente en cada test gracias al trait RefreshDatabase.

📘 Concepto: Trait RefreshDatabase

Un trait en PHP es un mecanismo de reutilización de código que permite incluir métodos en múltiples clases. RefreshDatabase ejecuta las migraciones antes de cada test y las revierte después, garantizando que cada prueba comience con una base de datos limpia y no afecte a las demás.

6Despliegue y CI/CD

6.1 ¿Qué es CI/CD?

CI/CD son las siglas de Continuous Integration / Continuous Deployment (Integración Continua / Despliegue Continuo).

  • Integración Continua (CI): Cada vez que un desarrollador sube código al repositorio, se ejecutan automáticamente las pruebas para verificar que no se rompió nada.
  • Despliegue Continuo (CD): Si las pruebas pasan, el código se despliega automáticamente en el servidor de producción.

6.2 Pipeline de GitHub Actions

El archivo .github/workflows/test.yml define un pipeline que se ejecuta automáticamente en cada push a la rama main:

# .github/workflows/test.yml
name: Tests

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup PHP
        uses: shivammathur/setup-php@v2
        with:
          php-version: '8.2'
          extensions: mbstring, pdo, pdo_sqlite
      - name: Install dependencies
        run: composer install --no-interaction
      - name: Run migrations
        run: php artisan migrate
        env:
          DB_CONNECTION: sqlite
          DB_DATABASE: ':memory:'
      - name: Run tests
        run: php artisan test

6.3 Flujo del pipeline

  1. Desarrollador hace push de código a main.
  2. GitHub Actions detecta el push y crea una máquina virtual limpia con Ubuntu.
  3. Instala PHP 8.2 y las extensiones necesarias.
  4. Descarga el código del repositorio.
  5. Ejecuta composer install para instalar dependencias.
  6. Crea la base de datos SQLite en memoria.
  7. Ejecuta las migraciones para crear las tablas.
  8. Corre los 6 tests de PHPUnit.
  9. Si todos pasan ✅, el código puede integrarse. Si alguno falla ❌, el merge queda bloqueado.
📘 Concepto: Merge bloqueado

Es una configuración de GitHub que impide que el código se integre a la rama principal si el pipeline de CI/CD no pasa. Esto garantiza que el código roto nunca llegue a producción. El equipo recibe una notificación indicando qué test falló y por qué.

7.2 Capa 1: Sanctum — Token de acceso

Cada petición privada necesita el header Authorization: Bearer {token}. Sin token válido, Laravel responde con HTTP 401 automáticamente (la petición nunca llega al controller).

Los tokens se almacenan hasheados en la tabla personal_access_tokens. Al cerrar sesión, el token se elimina de la base de datos y deja de funcionar instantáneamente.

📘 Concepto: Token Bearer

Es un string opaco (no legible) que identifica a un usuario autenticado. El cliente lo envía en cada petición en el header Authorization. El servidor lo valida contra la base de datos. Si coincide, identifica al usuario. Si no existe o fue revocado, rechaza la petición.

📘 Concepto: Sanctum vs JWT

JWT (JSON Web Token) es un token firmado digitalmente que contiene información del usuario. Su problema: no se puede revocar antes de que expire. Si un usuario cierra sesión, el JWT sigue siendo válido hasta su fecha de vencimiento. Sanctum almacena los tokens en la base de datos: al hacer logout se borran y dejan de funcionar de inmediato.

7.3 Capa 2: CheckRole — Verificación de rol

El middleware CheckRole verifica que el usuario autenticado tenga el rol requerido para acceder al endpoint:

  • role:cliente — puede reservar, no puede ver agendas de médicos.
  • role:profesional — solo ve y gestiona sus propios turnos.

Rol incorrecto → HTTP 403 Forbidden.

7.4 Capa 3: StoreTurnoRequest — Validación

Se ejecuta antes del controller. Verifica:

  • profesional_id: debe existir en la base de datos (rule: exists:users,id).
  • fecha_hora: debe ser en el futuro (rule: after:now) — no permite turnos en el pasado.

Dato inválido → HTTP 422 con mensaje en español indicando exactamente qué falló.

7.5 Capa 4: Autorización por recurso

En cambiarEstado(), el controller verifica que profesional_id === Auth::id(). No alcanza con tener rol 'profesional': hay que ser el dueño de ese turno específico.

7.6 Protecciones adicionales

  • $fillable: Lista blanca de campos que pueden ser asignados masivamente. Previene Mass Assignment Attack.
  • SoftDelete: Los turnos nunca se borran físicamente. El historial es permanente.
  • bcrypt: Las contraseñas se almacenan con Hash::make() que aplica bcrypt con salt automático.
  • Índice compuesto: (profesional_id, fecha_hora) optimiza la consulta de concurrencia.
📘 Concepto: Mass Assignment Attack

Es una vulnerabilidad donde un atacante envía campos adicionales en una petición HTTP para modificar campos que no debería poder cambiar. Por ejemplo, enviar rol: admin al registrarse. La propiedad $fillable define explícitamente qué campos pueden ser asignados masivamente, bloqueando cualquier campo no autorizado.

8.1 Puntos fuertes del proyecto
